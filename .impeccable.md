# Impeccable Design Context — Muscat Bay Utility Management

## Design Context

### Users
- **Primary**: Operations staff and facility/asset managers at Muscat Bay needing quick visibility on water, electricity, STP, assets, and contractors.
- **Secondary**: Executives who want high-level KPIs on the dashboard.
- **Context**: Mainly on-site via laptop/tablet in control rooms and offices. Occasionally mobile for field engineers/supervisors checking trends or alerts.
- **Job to be done**: Monitor utility operations, spot anomalies fast, track assets and contractors, and make data-driven decisions without switching between multiple tools.

### Brand Personality
- **Three words**: Professional, reliable, modern.
- **Voice & tone**: Serious operations tool — not a marketing site or gaming dashboard. Visual language closer to industrial/BMS dashboards (Siemens Desigo, Schneider EcoStruxure style).
- **Emotional goals**:
  - **Confidence & trust**: Data looks reliable, stable, and consistent across pages.
  - **Calm control**: Even during critical events, layout stays clean and structured, never chaotic.
  - **Low-key urgency**: Alarms use red badges and clear alert banners without overwhelming the rest of the UI.

### Aesthetic Direction
- **Visual tone**: Clean enterprise dashboard — flat, minimal, focused on data. Think Grafana, Azure Portal-style cards, modern BMS web HMIs.
- **Anti-references**: No playful neon themes, heavy gradients, overly "sales/marketing" aesthetics, or dense SCADA mimic panels with too many tiny icons.
- **Theme**: Dark mode is primary (control-room usage, long sessions, night shifts). Light mode is first-class for executives and daytime office use — both modes must be equally readable.
- **Color palette**:
  - Primary brand: `#4E4456` (purple-gray) — headers, sidebar, branding
  - Secondary/accent: `#00D2B3` (teal) — CTAs, active states, highlights
  - Status: green/blue = normal, amber/orange = warning, red = critical (used sparingly)
  - Avoid: very saturated neon colors, many different bright colors across cards
  - Cross-module consistency: water/electricity/STP cards should feel part of one unified system
- **Design tokens**: Follow and refine tokens in `DESIGN_SYSTEM.md`. Explicit tokens exist for status colors and chart palettes.

### Design Principles

1. **Data first, decoration never** — Every visual element must serve the data. No ornamental gradients, unnecessary illustrations, or distracting flourishes. If it doesn't help the operator understand status faster, remove it.

2. **Calm by default, urgent only when earned** — The UI should feel steady and controlled. Reserve red, animation, and elevated visual weight exclusively for genuine alarms. Normal state = quiet confidence.

3. **One system, many modules** — Water, electricity, STP, assets, and contractors are different domains but must feel like pages of the same application. Consistent card styles, spacing, typography, and interaction patterns across all modules.

4. **Accessible in the field** — Large touch targets for tablet/glove use. Never rely on color alone for status — always pair with icons (check, triangle, circle) and text labels. Meet WCAG AA contrast in both themes. Support reduced motion preferences. Plan for RTL (Arabic) layout.

5. **Equally readable in light and dark** — Both themes are production-grade. Dark mode biased for control rooms; light mode for executive/office use. Neither is an afterthought.

### Tech Stack Reference
- **Framework**: Next.js 16 + React 19 + TypeScript 5
- **Styling**: Tailwind CSS 4 with CSS variables in `app/globals.css`
- **UI primitives**: shadcn/ui (base-vega style) in `components/ui/`
- **Icons**: Lucide React — named imports from `"lucide-react"`
- **Charts**: Recharts 3 — components in `components/charts/`
- **Animations**: GSAP (subtle, minimal — respect reduced motion)
- **Utility**: `cn()` from `@/lib/utils` for conditional class merging (clsx + tailwind-merge)

### Accessibility Requirements
- **Level**: WCAG AA minimum on all key KPIs, buttons, and charts (especially dark mode)
- **Color blindness**: Never use red/green alone for status. Combine color with icons and short labels ("Critical", "Normal")
- **Touch targets**: Generous sizing and spacing for tablet/glove use on-site
- **Reduced motion**: Animations are subtle and non-essential. No auto-moving charts or flashing indicators for normal information
- **RTL support**: Direction-agnostic layout and components. Icon positions, chart legends, and nav placement should gracefully flip for Arabic UI

### Typography
- **Font**: System sans-serif stack (`ui-sans-serif, system-ui, sans-serif`)
- **Body**: 14px / 400 weight / 21px line-height
- **Headings**: 15-15.75px / 600-700 weight / semibold tracking-tight
- **Buttons**: 12.25-14px / 500 weight

### Spacing & Layout
- **Border radius**: `10.5px` (cards), `5px` (buttons, small elements)
- **Grid**: 4-column layout with 14px gap
- **Sidebar**: 208px expanded, 70px collapsed
- **Content max-width**: Responsive with `layout-shell` class
- **Transitions**: `0.15s cubic-bezier(0.4, 0, 0.2, 1)` — fast, professional
